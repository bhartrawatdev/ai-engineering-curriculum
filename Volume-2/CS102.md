## Course Description

CS102 builds the practical, professional-workflow layer that sits underneath every other course in the program. While CS101 teaches students *how to write* a program, CS102 teaches them how to *operate the environment* real software is built in: a Linux terminal, a filesystem they navigate deliberately, and Git as the version-control backbone of every future collaborative project. This course is intentionally tool-focused rather than theory-focused — it exists so that "how do I submit this?" or "how do I not lose my work?" is never a barrier to learning in any later course.

---

## Why This Course Is Needed

Every lab, assignment, and project from CS103 onward assumes command-line comfort and Git-based submission. Without this course, students would spend their entire first year re-learning tooling ad hoc, inconsistently, and under deadline pressure — exactly when it's hardest to learn well. Placing it in Semester 1, running concurrently with CS101, ensures tooling fluency exists *before* it's urgently needed.

## Prerequisite Knowledge Required

None. Designed to be accessible to complete beginners in parallel with CS101.

## Where It Is Used Later

- **CS101/CS103** — Git submission of the Week 14–15 mini-project (first graded use).
- **Every subsequent lab course** — all labs from Semester 2 onward assume terminal fluency and Git submission as the default workflow.
- **SE201 (Software Engineering Principles, Semester 4)** — formalizes branching strategies, pull requests, and code review that are only introduced informally here.
- **CS204 (Operating Systems, Semester 4)** — builds on the filesystem/process mental model introduced here, at a much deeper theoretical level.
- **PR301, PR401/PR402 (Semester Project I & Capstone)** — Git commit history and process discipline, graded criteria first introduced in this course.

---

## Learning Objectives

By the end of this course, students will be able to:

1. Navigate, inspect, and manipulate a Linux filesystem confidently from the command line.
2. Write and run simple shell scripts to automate repetitive tasks.
3. Use Git for local version control: init, add, commit, log, diff, branch, merge.
4. Collaborate on a shared GitHub repository using clone, push, pull, and pull requests.
5. Diagnose and resolve basic Git conflicts and common terminal errors independently.

## Weekly Topics / Units

| Week | Topic | Depth |
|---|---|---|
| 1 | What is Linux? Shell vs. terminal vs. OS. Setting up the environment | Conceptual + setup |
| 2 | Filesystem navigation: `pwd`, `ls`, `cd`, absolute vs. relative paths | Core skill |
| 3 | File & directory operations: `mkdir`, `touch`, `cp`, `mv`, `rm`, `rmdir` | Core skill |
| 4 | Viewing & editing files: `cat`, `less`, `head`, `tail`, intro to a terminal editor (nano/vim basics) | Core skill |
| 5 | Permissions & ownership: `chmod`, `chown`, `ls -l` interpretation | Core skill |
| 6 | Text processing basics: `grep`, `wc`, `sort`, pipes (`\|`) and redirection (`>`, `>>`, `<`) | Intermediate skill |
| 7 | Shell scripting I: variables, `.sh` files, running scripts, basic control flow in bash | Automation |
| 8 | Shell scripting II: loops, arguments, exit codes, simple automation scripts | Automation |
| 9 | Introduction to Git: what is version control, why it matters, `git init`, `git status` | Conceptual + core |
| 10 | Git local workflow: `add`, `commit`, `log`, `diff`, `.gitignore` | Core skill |
| 11 | Git branching: `branch`, `checkout`/`switch`, `merge`, understanding the commit graph | Core skill |
| 12 | GitHub & remote workflow: `clone`, `push`, `pull`, remotes, SSH vs. HTTPS auth | Collaboration |
| 13 | Pull requests & code review basics: opening a PR, reviewing, requesting changes, merging | Collaboration |
| 14 | Merge conflicts: causes, reading conflict markers, resolving manually | Troubleshooting |
| 15 | Integration week: end-to-end workflow simulation + review | Consolidation |

## Laboratory Syllabus (Summary — full detail in future Phase 5 pass)

- Weeks 1–8: individual terminal/scripting labs, escalating from guided to independent.
- Weeks 9–14: paired/group labs simulating real collaboration (shared repo, assigned PR reviews).
- Week 15: full workflow simulation — clone a repo, make a change on a branch, open a PR, resolve a seeded conflict, merge.

## Assignments (Summary — full detail in future Phase 6 pass)

- 3 assignments: (1) filesystem + scripting, (2) local Git workflow, (3) full GitHub collaboration workflow including a real pull request against a classmate's repo.

## Mini Projects

- **Capstone Lab (Week 15):** Each student contributes at least 2 meaningful commits and 1 pull request to a shared class repository (e.g., a crowd-sourced "useful shell scripts" collection), demonstrating the full local-to-remote-to-review pipeline.

## Semester Project Contribution

No standalone semester project — CS102's deliverable is the Week 15 collaborative Git exercise. However, its skills are the **submission mechanism** for every project from this point forward, starting with CS101's and CS103's mini-projects.
