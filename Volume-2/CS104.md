# CS104 — Object-Oriented Programming & Software Design

## Why It Matters / Where It's Used Later
CS101 taught you to write functions. CS104 teaches you to organize *state and behavior together* — the mental model behind almost every real codebase, library, and framework you'll touch from here on. Every ML library (scikit-learn's estimators, PyTorch's `nn.Module`) is built on OOP. Without this, CS105 (Data Structures) becomes syntax memorization instead of understanding, because every data structure you'll build is a class.

## Full Topic Breakdown

**1. Why OOP — problem it solves**
- Limits of pure functions + loose data (the "everything is a dict/list" ceiling)
- Data + behavior bundled together: the core idea

**2. Classes & Objects**
- `class` syntax, `__init__`, instance attributes vs. class attributes
- Creating and using instances
- `self` — what it actually is, not just convention

**3. Methods**
- Instance methods, `@staticmethod`, `@classmethod` — when to use each
- Special/dunder methods: `__str__`, `__repr__`, `__eq__`, `__len__`

**4. Encapsulation**
- Public vs. "private" (`_x`, `__x`) conventions in Python (no true private — why that's a deliberate language choice)
- Property decorators: `@property`, getters/setters done the Pythonic way

**5. Inheritance**
- Base classes, derived classes, `super()`
- Method overriding
- Multiple inheritance and MRO (Method Resolution Order) — just enough to not be surprised by it

**6. Polymorphism**
- Duck typing (Python's real philosophy vs. formal polymorphism)
- Abstract base classes (`abc` module), when to enforce an interface

**7. Composition vs. Inheritance**
- "Favor composition over inheritance" — why, with a concrete before/after example
- When inheritance is actually the right call

**8. Design Patterns (intro, practical subset only)**
- Factory pattern, Strategy pattern, Observer pattern — the 3 you'll actually see in real code
- Skip the rest (Gang of Four's other 20 patterns) until you hit a real need for them

**9. SOLID Principles (lightweight intro)**
- Single Responsibility and Open/Closed — the two that matter most day-to-day
- Others (Liskov, Interface Segregation, Dependency Inversion) — mentioned, not drilled

## Practice Exercises + Project

**Exercises:**
- Model a `BankAccount` class with deposit/withdraw and balance validation (encapsulation practice)
- Build a small class hierarchy: `Shape` → `Circle`, `Rectangle`, `Triangle` with a shared `area()` method (inheritance + polymorphism)
- Implement `__eq__` and `__repr__` for a custom class and prove they work with `==` and `print()`

**Project — "Library Management System" (CLI, no DB yet):**
Build a small in-memory library system: `Book`, `Member`, `Library` classes. Members can borrow/return books; library tracks availability and overdue status. Must use inheritance (e.g., `EBook` extends `Book`), composition (`Library` *has* books and members, doesn't inherit from them), and at least one design pattern (e.g., Factory for creating different book types).
